Project Nexus - Detailed Data Flow Architecture

Detailed draft architecture for a MySQL-first curated analytics platform acting as the Project Nexus data backbone. This draft assumes Governance-hosted reporting surfaces, Data Lake-controlled certified snapshots, GitLab-coupled documentation and pipelines, and a strict separation between generic Lake Functions and specialist module logic.

Architecture status: Draft Platform: MySQL + Python + GitLab Pattern: curated analytics backbone Integration: hub-and-spoke

1. Architecture position

The platform should be described honestly as a MySQL-first curated analytics platform, not as a full cloud-native lakehouse. It still behaves like a Data Lake operationally because it centralises ingestion, standardisation, MDM, lineage, access control, quality checks, reusable data products, and reporting snapshots.

This positioning matters because it prevents overpromising. The immediate goal is to make Project Nexus operable, auditable, and maintainable without prematurely forcing an Azure-first lakehouse before the scale and concurrency justify it.

2. Recommended system design

Storage

MySQL schemas for raw, standardised, curated, reference, audit, and reporting snapshot layers.

Compute

Python jobs for extraction, standardisation, validation, promotion, snapshotting, and dataset publication.

Control

GitLab repository for SQL, Python, mappings, MDM definitions, Wiki pages, changelogs, and release tags.

Consumption

Governance-facing reporting and compliance surfaces; module-specific consumers read standardised or curated datasets.

2.5 Where the modules sit

The modules also sit on the MySQL platform, but not as part of the shared Lake layer. Each module has its own MySQL schema or database for specialist logic, working tables, run outputs, and module-owned state, while the shared Data Lake schemas hold certified cross-module datasets, MDM, reference data, and immutable snapshots.

This gives one operational stack with clear separation: the Data Lake prepares and certifies data; modules crunch it; Governance consumes it. That matches the project boundary in the source document, where the Data Lake is the integration backbone and module logic remains specialist, module-owned work.

2.6 Why not a true lakehouse

The immediate problem is not primarily storage scale; it is governance, traceability, and controlled cross-module reuse. A curated MySQL-first platform gives us the fastest path to a reliable operating model, while still preserving the option to move to Azure when storage/compute separation, semi-structured data volume, or orchestration complexity actually justify it.

1. Structured first

The work is dominated by canonical entities, reporting-ready facts, and governed reference data, not broad object-storage landing of semi-structured files.

2. Control first

The system needs MDM, lineage, reconciliation, and audit-stamped snapshots before it needs distributed lakehouse-style compute separation.

3. Faster delivery

MySQL-first is quicker to stand up, easier to maintain, and easier for a small team to reason about than a cloud-native lakehouse from day one.

4. Lower burden

Operational burden stays manageable while the team still has room to tolerate manual scheduling and Python-based pipelines.

5. Migration path stays open

When volume, concurrency, or object-like storage needs become real constraints, Azure becomes a justified evolution rather than a premature default.

4. End-to-end flow

Step 1

Source ingestion

Core Admin DB data is replicated one-way into Raw. External feeds such as market data, investment data, regulatory templates, expense files, and reference files are also loaded into Raw. Module outputs that must be shared downstream are landed in controlled intake tables.

Step 2

Standardisation and conformance

Raw inputs are typed, validated, deduplicated, code lists are harmonised, identifiers are normalised, and domain-level conformance rules are applied. This is where source-system noise is removed without injecting specialist actuarial or accounting meaning.

Step 3

MDM and curated publication

Mastered entities are assembled, reference versions applied, lineage linked, and reusable datasets are published into Curated. These become the certified inputs for Cashfluid, Sun 2.x, Moody's 2.x, HKRBC Reporting, Transactional, and Governance.

Step 4

Module computation

Each module applies specialist logic to curated inputs. Outputs that matter cross-module are returned to the Data Lake through controlled interfaces rather than point-to-point links.

Step 5

Governance consumption and reporting

Governance reads certified curated outputs and reporting snapshots to drive board packs, compliance workflows, attestations, and reporting access. Published reporting outputs are backed by immutable Data Lake snapshots so every figure can be reproduced.

6. Reporting marts stance

Reporting marts should live logically in Governance because they are consumption products tied to approval, oversight, attestation, and board or regulatory dissemination. However, the exact data behind each published output should remain materialised in the Data Lake as immutable reporting snapshots.

This creates a clean split: Governance owns the front-end and workflow context; the Data Lake owns the certified, versioned, reproducible data behind every published result.

7. Lake Functions vs Module Functions

Lake Functions

  • Ingestion and intake control
  • Standardisation and deduplication
  • Code-list harmonisation
  • MDM survivorship
  • Reference-data versioning
  • Snapshotting
  • Lineage capture
  • Data quality controls
  • Access governance and masking
  • Generic reusable aggregations only

Module Functions

  • Liability cash-flow projection
  • Journal validation and posting-rule logic
  • CSM roll-forward and disclosure assembly
  • PCR, MCR, BEL, MOCE, TVOG capital logic
  • Transactional carve-out and deal logic
  • Governance attestation and workflow logic

8. Pipeline and orchestration approach

Given the tolerance for operational burden, the first release can use scheduled Python jobs with disciplined logging, run IDs, and idempotent load patterns. Each job should support reruns, partial backfills, and replay using extract cutoffs, source batch IDs, or effective dates.

Move to an orchestrator only when dependency chaining, retries, environment promotion, or monitoring become too fragile to manage manually. That means cron or equivalent first, orchestrator later.

9. Controls, validation, and recovery

Control areaRecommendation
IdempotencyAll ingestion and publication jobs should be rerunnable without duplicating results.
BackfillsSupport date-range or batch-range reprocessing with versioned outputs.
LoggingCapture run ID, source cut, row counts, validation results, and release tag.
Data validationApply row-count, null, uniqueness, referential, and reconciliation checks before promotion.
AlertingTrigger alerts on failed loads, reconciliation breaks, stale datasets, and unauthorized schema drift.
RecoveryRetain raw loads, change versions, and snapshots so a curated release can be rebuilt exactly.

10. Security and access

  • Separate technical write access from analyst read access.
  • Mask PII in non-privileged views and datasets.
  • Restrict sensitive domains by role, domain, and business need.
  • Log data access to sensitive datasets and report snapshots.

11. What to build first

  1. Raw and standardised ingestion path from Core Admin DB and the highest-priority external feeds.
  2. First-wave MDM domains: Customer, Policy, Product, CoA mapping, Currency, Calendar.
  3. Lineage, quality, and snapshot controls before adding too many downstream consumers.
  4. Certified curated datasets for the most urgent module consumers.
  5. Governance-facing reporting snapshot handoff and audit trail model.

14. Consistency check

The architecture and MDM drafts now use the same operating logic: MySQL-first, module-owned crunching, Lake-owned certification and control, Governance-owned consumption and oversight, and an Azure migration only when scale or orchestration complexity justifies it.

12. What to postpone

  • Azure-native lakehouse tooling unless MySQL storage, query concurrency, or binary/object-scale storage become painful.
  • Heavy orchestration tools until cron-style scheduling becomes operationally unsafe.
  • Broad document repository functions inside MySQL beyond metadata and structured tracking.
  • Overly ambitious real-time streaming unless a real freshness need appears.
