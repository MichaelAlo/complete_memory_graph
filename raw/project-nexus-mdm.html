Project Nexus - Draft Master Data Management Plan

Drafted for the Data Lake and Data Wiki workstream as a MySQL-first curated analytics platform. This draft assumes the Data Lake is the single integration backbone, Governance is the likely reporting access layer, and published reporting outputs must remain reconcilable to auditable snapshots held in the Data Lake.

Status: Draft Owner: Michael Alo Platform stance: MySQL-first curated analytics platform Reporting stance: Governance-facing, Lake-backed snapshots

1. Relationship to architecture

This MDM is the control layer that makes the architecture operational. It defines what is mastered, how versions are governed, how changes are approved, and how curated datasets stay reproducible across modules and reporting outputs.

1. Summary recommendation

The immediate target is not a true lakehouse. It is a curated analytics platform that behaves like a controlled Data Lake operationally: one-way ingestion from source systems, standardised and versioned curated layers, reusable Lake Functions, lineage, MDM, access control, and auditable reporting snapshots.

In this model, MySQL stores structured data, reference data, master data, reconciled snapshots, and reporting-support datasets. Governance consumes certified outputs for reporting and compliance workflows, while the Data Lake remains the integration and control backbone.

2. Why this is not a true lakehouse yet

The current problem is not primarily storage scale; it is governance, traceability, and controlled cross-module reuse. A curated MySQL-first platform gives us the fastest path to a reliable operating model, while still preserving the option to move to Azure when storage/compute separation, semi-structured data volume, or orchestration complexity actually justify it.

This MDM is therefore written for a controlled Data Lake operational model, not for a broad lakehouse. It focuses on clear masters, versioned reference data, reproducible history, and explicit approval workflows.

2. Scope and purpose

Purpose

Define how canonical business entities are governed, versioned, documented, changed, and consumed across Project Nexus.

In scope

Canonical entities, reference data, source ownership, quality rules, effective dating, lineage, approvals, access control, reconciliation, and change management.

Out of scope

Specialist actuarial, accounting, HKFRS17, HKRBC, transactional, and Governance workflow logic. Those remain Module Functions unless later proven generic.

3. Design assumptions

  • The Data Lake is the hub-and-spoke backbone for all inter-system exchange.
  • The Core Admin DB remains outside Nexus and feeds the platform one-way only.
  • The near-term platform is MySQL-first, with Python routines and GitLab-backed documentation and change control.
  • Reporting marts are likely to sit in Governance, but the underlying published reporting snapshots must remain persisted in the Data Lake.
  • Lake Functions are limited to generic reusable data-engineering and governance controls.

5. Canonical model rules

  • Each mastered entity must have a canonical identifier, business key, source-system key set, status, effective-from, effective-to, created timestamp, updated timestamp, and provenance fields.
  • Every attribute must be tagged as required, optional, derived, reference-only, or sensitive.
  • Sensitive fields must be classified for PDPO handling, masking rules, and access restrictions.
  • Historical states must be queryable. Type 2-style versioning or equivalent effective-dated history is required for mastered entities.

7. Recommended Lake Functions registry

Recommend now

Ingestion, standardisation, deduplication, MDM survivorship, reference data versioning, code-list harmonisation, snapshotting, lineage capture, access governance, and data quality controls.

Do not centralise now

Liability cash-flow projection, journal validation against specialist posting rules, CSM roll-forward logic, HKRBC capital calculations, transaction modelling, and Governance attestation workflow.

Tie-break rule

If the logic is generic and reusable, keep it in the Lake. If it depends on actuarial, accounting, regulatory, or transaction-specific meaning, keep it in the relevant module.

9. Change-control workflow linked to MDM

  1. Raise change request: every schema, semantic, mapping, quality-rule, masking-rule, or reporting-methodology change starts as a GitLab issue.
  2. Assess impact: identify affected domains, datasets, modules, reports, backfills, controls, and approvals.
  3. Update artefacts together: MDM entry, Data Wiki page, lineage metadata, quality rules, and Lake Functions Registry must be updated in the same change set.
  4. Approve: domain owner, affected module owner, Data Lake owner, and Governance reviewer if compliance or reporting is affected.
  5. Implement: deploy SQL, Python, mapping, and documentation changes as one tagged release.
  6. Validate: run reconciliation checks, regression comparisons, and required human-in-the-loop sign-off.
  7. Version and publish: mark effective-from dates, retain prior versions, and publish a changelog entry in the Wiki.
Control rule: no data-methodology change is complete until code, Wiki, MDM, lineage metadata, and approvals are all updated in the same controlled release.

10. Data quality and reconciliation

  • Each curated dataset must have named quality checks for completeness, uniqueness, validity, referential integrity, and timeliness.
  • Every key published metric must reconcile backward to the source system and forward to consuming reports.
  • Breaks must be logged, triaged, assigned, and closed with root cause and corrective action.
  • Report snapshots must store the extract timestamp, source cut, transformation version, approver, and release identifier.

11. Wiki maintenance model

The Data Wiki should be coupled to the GitLab repository used for pipeline code and MDM definitions. Each major domain, Lake Function, source system, and curated dataset should have a dedicated page with ownership, refresh cadence, schema summary, quality rules, downstream consumers, and change history.

The Coreflexi wiki can remain separate, but the Data Lake wiki should maintain stable references to it and surface any upstream changes that alter downstream assumptions.
